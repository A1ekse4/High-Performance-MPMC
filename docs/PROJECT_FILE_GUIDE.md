# Подробный разбор файлов проекта `High-Performance-MPMC`

## Что это за проект

Этот репозиторий показывает полный маленький исследовательский цикл вокруг lock-free структуры данных:

1. Реализуется bounded MPMC-очередь на атомиках.
2. Проверяется корректность в тестах.
3. Измеряется производительность в бенчмарке.
4. Собираются профили `perf`.
5. Пишутся CSV и строятся графики для анализа bottleneck'ов.

Проект небольшой по размеру, но в нем есть почти все типичные слои системного performance-oriented проекта:

- ядро алгоритма;
- тесты на корректность;
- нагрузочный запуск;
- автоматизация серий запусков;
- визуализация результатов;
- профилирование и агрегация профилей.

## Быстрая карта проекта

- `CMakeLists.txt` - сборка библиотеки, тестов и бенчмарка.
- `README.md` - пользовательское описание проекта, сценарии запуска и интерпретация результатов.
- `include/mpmc_bounded_queue.hpp` - основная реализация lock-free bounded MPMC-очереди.
- `tests/test_mpmc.cpp` - проверки FIFO, отсутствия потерь/дубликатов и прогресса под contention.
- `bench/bench_mpmc.cpp` - исполняемый микробенчмарк.
- `scripts/run_dense_bench.py` - массовый прогон матрицы сценариев `P x C`.
- `scripts/plot_benchmarks.py` - построение benchmark-графиков по CSV.
- `scripts/profile.sh` - запуск одного профилирования через `perf`.
- `scripts/plot_profiling_runs.py` - агрегация многих профилирований и построение summary-графиков.
- `docs/images/profiling_runs_summary.csv` - пример агрегированной таблицы метрик профилирования.
- `docs/images/profiling_symbol_distribution_table.csv` - пример таблицы распределения CPU samples по категориям кода.
- `.gitignore` - исключения для git.

## Как части проекта связаны друг с другом

Типичный поток такой:

1. `CMakeLists.txt` собирает `mpmc_tests` и `bench_mpmc`.
2. `tests/test_mpmc.cpp` использует `include/mpmc_bounded_queue.hpp`, чтобы проверить корректность.
3. `bench/bench_mpmc.cpp` использует ту же очередь, но уже для измерения throughput.
4. `scripts/run_dense_bench.py` много раз вызывает `build/bench_mpmc` и складывает результаты в CSV.
5. `scripts/plot_benchmarks.py` берет benchmark CSV и строит графики.
6. `scripts/profile.sh` вызывает `bench_mpmc`, затем `perf stat` и `perf record`.
7. `scripts/plot_profiling_runs.py` читает накопленные profiling run'ы, агрегирует метрики и строит сводные артефакты.
8. `README.md` связывает все это в единый пользовательский сценарий.

## `CMakeLists.txt`

### Назначение

Главный файл сборки. Он описывает, как проект компилируется и какие бинарники создает.

### Что происходит внутри

- Задается минимальная версия CMake: `3.20`.
- Имя проекта: `HighPerformanceMPMC`.
- Язык проекта: `CXX`.
- Стандарт C++: `C++20`.
- Отключаются compiler-specific extensions через `CMAKE_CXX_EXTENSIONS OFF`, чтобы код был ближе к стандартному C++.

### Важные части

- `option(MPMC_ENABLE_NATIVE ...)`
  Позволяет включать `-march=native`. Это полезно для локального performance-тюнинга, но ухудшает переносимость бинарника между CPU.

- `find_package(Threads REQUIRED)`
  Подтягивает системную поддержку потоков, потому что и тесты, и бенчмарк используют `std::thread`.

- `add_library(mpmc_queue INTERFACE)`
  Библиотека тут header-only. У нее нет `.cpp`-реализации: весь код лежит в `include/mpmc_bounded_queue.hpp`.

- `target_include_directories(mpmc_queue INTERFACE include)`
  Делает директорию `include/` доступной для всех целей, которые линкуются с `mpmc_queue`.

- `add_executable(mpmc_tests tests/test_mpmc.cpp)`
  Создает тестовый бинарник.

- `add_executable(bench_mpmc bench/bench_mpmc.cpp)`
  Создает бинарник для измерения производительности.

- `enable_testing()` и `add_test(...)`
  Подключают `ctest`, чтобы тесты можно было запускать стандартным способом.

### Роль в архитектуре

Это точка входа для всего build pipeline. Без нее проект не соберется, а скрипты не смогут найти `build/bench_mpmc`.

## `include/mpmc_bounded_queue.hpp`

### Назначение

Это главный файл проекта. В нем находится вся реализация bounded lock-free MPMC-очереди `BoundedMPMCQueue<T>`.

### Общая идея алгоритма

Очередь построена вокруг кольцевого буфера фиксированного размера. Размер должен быть степенью двойки, чтобы индексирование можно было делать через маску:

- физическая ячейка: `buffer_[pos & mask_]`;
- `mask_ = capacity - 1`.

Каждая ячейка буфера хранит:

- `sequence` - номер состояния ячейки;
- сырой буфер `storage` для объекта `T`.

Алгоритм опирается на sequence-based protocol:

- producer смотрит, свободна ли ячейка для позиции `pos`;
- consumer смотрит, готова ли ячейка для чтения из позиции `pos`;
- после записи и чтения `sequence` переводится в следующее состояние.

Это классическая bounded MPMC-схема, близкая по духу к алгоритму Дмитрия Вьюкова.

### Пространство имен `detail`

#### `kCacheLineSize`

Константа `64`. Используется для выравнивания структур на cache line и снижения false sharing.

#### `is_power_of_two(std::size_t value)`

Проверяет, что capacity:

- не равен нулю;
- является степенью двойки.

Нужно для корректной работы `mask_`.

#### `cpu_relax()`

Функция для короткой "вежливой" паузы в spin-loop:

- на x86 вызывает `__builtin_ia32_pause()`;
- на других архитектурах делает `atomic_signal_fence`.

Это снижает агрессивность busy-wait цикла при contention.

### Класс `BoundedMPMCQueue<T>`

#### Конструктор `BoundedMPMCQueue(std::size_t capacity)`

Что делает:

- сохраняет `capacity_`;
- вычисляет `mask_ = capacity - 1`;
- выделяет `buffer_` нужного размера;
- проверяет, что capacity - степень двойки;
- инициализирует `sequence` каждой ячейки значением ее индекса.

Смысл начальной инициализации:

- ячейка `i` считается свободной для producer'а, который пытается занять логическую позицию `pos == i`.

#### Запрещенные copy-операции

- `BoundedMPMCQueue(const BoundedMPMCQueue&) = delete;`
- `operator=(...) = delete;`

Это разумно: структура содержит атомики, внутреннее состояние кольцевого буфера и вообще концептуально не предназначена для копирования.

#### Деструктор `~BoundedMPMCQueue()`

Вызывает `drain_remaining()`.

Это важно для типов `T`, у которых есть нетривиальный деструктор. Если в очереди на момент уничтожения остались элементы, их нужно корректно разрушить вручную, потому что объекты создаются placement new в сыром буфере.

#### `bool try_enqueue(const T& value)`

Перегрузка для lvalue. Просто перенаправляет вызов в `emplace_impl(value)`.

#### `bool try_enqueue(T&& value)`

Перегрузка для rvalue. Позволяет перемещать значение в очередь без лишнего копирования.

#### `template <typename... Args> bool try_emplace(Args&&... args)`

Позволяет конструировать объект `T` прямо внутри ячейки буфера через perfect forwarding.

Практический смысл:

- меньше временных объектов;
- более универсальный интерфейс для произвольных типов `T`.

#### `bool try_dequeue(T& out)`

Это основной consumer fast path.

Логика по шагам:

1. Читается текущая логическая позиция `dequeue_pos_`.
2. Выбирается ячейка `buffer_[pos & mask_]`.
3. Читается `cell.sequence` с `memory_order_acquire`.
4. Вычисляется `diff = seq - (pos + 1)`.

Интерпретация `diff`:

- `diff == 0`:
  ячейка готова к чтению, consumer может попытаться захватить позицию.
- `diff < 0`:
  данных для этой позиции еще нет, очередь на этом месте пуста.
- `diff > 0`:
  кто-то другой уже продвинул состояние, нужно перечитать глобальную позицию.

Если `diff == 0`, consumer делает `compare_exchange_weak` на `dequeue_pos_`:

- успех означает, что именно этот поток получил право вычитать элемент;
- после этого вызывается `move_out_cell(cell, out)`;
- затем `sequence` переводится в `pos + capacity_`, тем самым ячейка помечается как снова свободная для будущего producer'а после полного оборота кольца.

#### `std::size_t capacity() const noexcept`

Просто возвращает емкость очереди.

#### `std::size_t approx_size() const noexcept`

Возвращает приближенную оценку размера как разницу между `enqueue_pos_` и `dequeue_pos_`.

Почему это только approximate:

- между двумя чтениями другие потоки могут продвинуть обе позиции;
- метод не синхронизируется так, чтобы давать точный снапшот.

### Внутренние структуры

#### `Backoff`

Минимальная стратегия backoff:

- содержит счетчик `count`;
- пока `count < 16`, вызывает `cpu_relax()` и увеличивает счетчик.

Это очень легкий throttling в spin-loop. Здесь нет экспоненциального сна, yield или sleep; только короткий pause.

#### `Cell`

Одна ячейка кольцевого буфера:

- `std::atomic<std::size_t> sequence`;
- `std::byte storage[sizeof(T)]`.

`alignas(64)` уменьшает шанс, что соседние горячие ячейки будут биться за одну cache line.

#### `AlignedAtomicSizeT`

Обертка над `std::atomic<std::size_t>` с добивкой padding до размера cache line.

Используется для:

- `enqueue_pos_`;
- `dequeue_pos_`.

Смысл:

- producer'ы постоянно пишут в `enqueue_pos_`;
- consumer'ы постоянно пишут в `dequeue_pos_`;
- если они лежат на одной cache line, contention станет хуже из-за false sharing.

### Внутренние методы

#### `emplace_impl(Args&&... args)`

Основной producer fast path.

Логика симметрична `try_dequeue`:

1. Читается `enqueue_pos_`.
2. Берется ячейка `buffer_[pos & mask_]`.
3. Читается `cell.sequence`.
4. Вычисляется `diff = seq - pos`.

Интерпретация:

- `diff == 0`:
  ячейка свободна для записи именно в эту логическую позицию.
- `diff < 0`:
  очередь переполнена на этой позиции, producer не может продвинуться.
- `diff > 0`:
  кто-то другой уже двинулся вперед, нужно перечитать `enqueue_pos_`.

При успешном `compare_exchange_weak`:

- объект конструируется в ячейке через `construct_in_cell(...)`;
- затем `sequence` переводится в `pos + 1`, то есть ячейка становится "готовой для чтения".

#### `construct_in_cell(Cell& cell, Args&&... args)`

Использует placement new:

- объект `T` создается прямо в `cell.storage`.

Это необходимо, потому что буфер не хранит `T` как обычный готовый массив объектов; он хранит сырую память.

#### `move_out_cell(Cell& cell, T& out)`

Что делает:

- получает указатель на объект `T` через `reinterpret_cast` + `std::launder`;
- перемещает значение в `out`;
- если `T` не trivially destructible, вручную вызывает деструктор.

Это важный метод жизненного цикла объекта в lock-free контейнере.

#### `drain_remaining()`

Используется только в деструкторе и только для нетривиально разрушаемых типов.

Что делает:

- читает диапазон между `dequeue_pos_` и `enqueue_pos_`;
- для каждой потенциально занятой позиции проверяет `sequence == pos + 1`;
- если объект действительно присутствует, вызывает его деструктор вручную.

Это нужно, чтобы не утекали ресурсы объектов, оставшихся в очереди.

### Memory ordering

В файле очень важно понимать не только "что", но и "как":

- `sequence.load(... acquire)` на producer/consumer пути гарантирует, что после наблюдения готового состояния поток увидит и сами данные объекта.
- `sequence.store(... release)` публикует новое состояние ячейки после записи или освобождения.
- `compare_exchange_weak(... relaxed, relaxed)` на позициях используется только для захвата индекса; публикация содержимого идет через `sequence`.

То есть протокол синхронизации строится не на глобальных позициях, а на `sequence` конкретной ячейки.

### Что реализовано хорошо

- bounded lock-free queue без mutex в fast path;
- поддержка move/emplace;
- попытка уменьшить false sharing;
- корректный manual lifetime management для нетривиальных типов;
- простая и понятная схема backoff.

### Ограничения и особенности

- интерфейс `try_*`, а не blocking API;
- очередь bounded, размер нельзя расширить;
- `approx_size()` не дает точного значения;
- progress на практике lock-free, но не wait-free;
- для пользователя важно самому решать, что делать при `false` из `try_enqueue`/`try_dequeue`.

## `tests/test_mpmc.cpp`

### Назначение

Это файл с ручными self-check тестами на `assert`, без внешнего test framework.

### Общая структура

Внутри анонимного namespace лежат три тестовые функции, а в `main()` они вызываются по очереди.

### `test_single_thread_fifo()`

Проверяет базовую FIFO-корректность в одном потоке.

Что делает:

- создает очередь на `1024`;
- кладет `1000` чисел от `0` до `999`;
- затем читает их обратно;
- проверяет, что порядок совпадает;
- проверяет, что после исчерпания очередь пуста.

Что именно этот тест доказывает:

- базовая логика кольцевого буфера работает;
- нет нарушения порядка в простом случае;
- пустая очередь корректно возвращает `false`.

### `test_mpmc_no_loss_no_duplicates()`

Это главный тест корректности под реальной конкуренцией.

Параметры:

- `4` producer;
- `4` consumer;
- `100000` элементов на producer.

Идея проверки:

- каждый producer генерирует уникальный диапазон идентификаторов;
- consumers параллельно вычитывают элементы;
- массив `seen` хранит счетчик, сколько раз встретился каждый id.

Что считается успехом:

- каждый id встретился ровно один раз;
- суммарно было получено `total_items`.

Что этот тест проверяет:

- нет потери элементов;
- нет дубликатов;
- нет "размазывания" значений между потоками;
- очередь выдерживает типичный конкурентный сценарий.

### `test_progress_under_contention()`

Тест не на строгую функциональную корректность, а на живость и прогресс.

Сценарий:

- `8` producer и `8` consumer;
- queue capacity `1 << 12`;
- потоки работают `2` секунды;
- считают число успешных `push` и `pop`.

В конце проверяется:

- `pushed > 1000`;
- `popped > 1000`.

Это не математическое доказательство lock-freedom, но хороший smoke test на то, что под заметным contention система не "умирает".

### `main()`

Последовательно вызывает все три теста и печатает:

- `All MPMC queue tests passed`

### Практическая ценность файла

Этот файл закрывает три уровня риска:

- логическая FIFO-ошибка;
- ошибка конкурентной корректности;
- отсутствие прогресса под нагрузкой.

## `bench/bench_mpmc.cpp`

### Назначение

Это исполняемый микробенчмарк для измерения throughput очереди.

### Структура файла

В анонимном namespace находятся:

- структура `Result`;
- функция `run_case(...)`;
- функция `make_thread_matrix()`.

Затем идет `main(...)`.

### `struct Result`

Содержит результаты одного benchmark-case:

- `producers`;
- `consumers`;
- `capacity`;
- `duration_seconds`;
- `enq_ops`;
- `deq_ops`.

Это минимальный набор для дальнейшего расчета effective throughput.

### `run_case(...)`

Главная функция измерения.

Аргументы:

- число producer threads;
- число consumer threads;
- capacity;
- длительность прогона.

Что делает:

1. Создает очередь.
2. Создает флаги:
   - `start` - барьер старта;
   - `stop` - флаг остановки.
3. Поднимает producer threads.
4. Поднимает consumer threads.
5. Все потоки ждут `start`.
6. После старта работают до `stop`.
7. Каждый поток сохраняет локально число успешных операций.
8. После join результаты суммируются и возвращаются.

Почему локальные счетчики хороши:

- нет лишнего shared atomic contention на глобальном счетчике операций;
- меньше искажений результата самого benchmark.

Почему используется `yield()` на неуспешной попытке:

- бенчмарк не пытается измерить абсолютно "голый" tight-spin;
- немного снижает агрессивность при пустой/полной очереди.

### `make_thread_matrix()`

Строит набор количеств потоков для автоматического sweep:

- `1, 2, 4, 8, ...` пока не дойдет до `hardware_concurrency()`;
- если последнее значение не равно `hw`, оно добавляется отдельно.

Это дает быстрый scaling sweep без полного квадрата от `1` до `hw`.

### `main(int argc, char** argv)`

Поддерживает два режима.

#### Режим 1: single-case

Если аргументов ровно 4 пользовательских:

```text
bench_mpmc <producers> <consumers> <capacity> <seconds>
```

Тогда:

- запускается ровно один кейс;
- печатается CSV-заголовок;
- печатается одна строка результата.

Это режим, который особенно важен для `perf` и скриптов автоматизации.

#### Режим 2: built-in matrix mode

Если аргументов нет:

- создается матрица потоков через `make_thread_matrix()`;
- capacities фиксированы: `4096`, `32768`, `262144`;
- каждый кейс длится `2` секунды;
- все результаты печатаются в stdout как CSV.

### Формула throughput

Используется:

- `effective_ops_per_sec = min(enq_ops, deq_ops) / duration`

Это разумно, потому что реальная "сквозная" throughput MPMC-очереди ограничена более слабой стороной:

- если enqueue сильно больше dequeue, то полный pipeline не может считаться по enqueue;
- и наоборот.

### Роль файла в проекте

Это основной генератор численных performance-данных. Без него не работают:

- dense benchmark sweep;
- profiling;
- построение графиков.

## `scripts/run_dense_bench.py`

### Назначение

Автоматизирует полный перебор матрицы сценариев benchmark'а.

Если `bench_mpmc` - это низкоуровневый генератор одного результата, то этот скрипт - orchestration layer для серии запусков.

### `run_case(...)`

Что делает:

- формирует команду запуска `bench_mpmc`;
- выполняет ее через `subprocess.run(..., check=True, capture_output=True, text=True)`;
- берет stdout;
- ожидает минимум две непустые строки:
  - CSV header;
  - CSV data row;
- собирает словарь `header -> value`.

Если формат вывода неожиданно отличается, бросает `RuntimeError`.

То есть функция не просто запускает бинарник, а одновременно валидирует протокол его вывода.

### `main()`

Разбирает CLI-аргументы:

- путь к бинарнику;
- диапазон producers;
- диапазон consumers;
- одна capacity или список capacities;
- длительность;
- путь выходного CSV.

Дальше:

1. Проверяет существование `bench_bin`.
2. Создает родительскую директорию для output.
3. Парсит capacities:
   - либо одно значение `--capacity`;
   - либо CSV-строка из `--capacities`.
4. Считает общее число запусков `total`.
5. Тройным циклом проходит по:
   - capacity;
   - producers;
   - consumers.
6. Каждый результат добавляет в `rows`.
7. Печатает прогресс с throughput в `Mops/s`.
8. В конце пишет финальный CSV.

### Почему этот файл полезен

Он переводит проект из режима "один ручной запуск" в режим "получить полную поверхность производительности".

### Формат результата

Скрипт пишет CSV с колонками:

- `producers`
- `consumers`
- `capacity`
- `enq_ops`
- `deq_ops`
- `effective_ops_per_sec`

Именно этот CSV потом читает `scripts/plot_benchmarks.py`.

## `scripts/plot_benchmarks.py`

### Назначение

Строит визуализации по benchmark CSV.

Это аналитический слой поверх `run_dense_bench.py`.

### Особенность

Сразу переключает backend matplotlib в `Agg`, то есть скрипт может работать headless, без GUI.

### `@dataclass BenchRow`

Типизированное представление строки CSV:

- упрощает работу с полями;
- делает код читаемее, чем постоянное обращение к сырым словарям.

### `read_rows(path)`

Читает CSV и превращает каждую строку в `BenchRow`.

Это обычный слой парсинга, который из текстового формата делает структурированные объекты.

### `plot_heatmaps(rows, output_dir)`

Строит тепловые карты throughput по всем capacity.

Идея:

- отдельный subplot на каждую capacity;
- ось X - consumers;
- ось Y - producers;
- значение клетки - `effective_ops_per_sec` в M ops/sec.

Важные детали:

- если сценарий отсутствует, подставляется `0.0`;
- в каждую клетку вписывается текстовое значение;
- рядом добавляется colorbar.

Это главный график для быстрого ответа на вопрос:

- где очередь масштабируется лучше;
- при каких соотношениях producer/consumer начинается деградация.

### `plot_top_scenarios(rows, output_dir, top_n=10)`

Сортирует все сценарии по throughput и показывает топ-N столбчатой диаграммой.

Полезно, когда нужно быстро понять:

- какие конфигурации вообще оказались лучшими;
- какие capacity и пары `P/C` выиграли глобально.

### `plot_capacity_trends(rows, output_dir)`

Строит кривые только для balanced-конфигураций `P == C`.

Для каждого значения `P=C` показывает, как меняется throughput при росте capacity.

Смысл:

- отделить влияние размера буфера от влияния асимметрии producer/consumer;
- увидеть, помогает ли больший буфер или нет.

### `main()`

Что делает:

- читает аргументы;
- создает output directory;
- загружает строки CSV;
- проверяет, что данные вообще есть;
- строит три типа графиков;
- печатает список созданных файлов.

### Что создает

- `throughput_heatmaps.png`
- `top_scenarios.png`
- `capacity_trends_balanced.png`

## `scripts/profile.sh`

### Назначение

Shell-обвязка для одного profiling-run. Это мост между benchmark-бинарником и инструментами `perf`.

### Что делает по шагам

1. Включает строгий режим bash:
   - `set -euo pipefail`
2. Определяет корень проекта и build/output директории.
3. Создает уникальный `RUN_ID` по времени.
4. Создает директорию `build/profiling/runs/<RUN_ID>`.
5. Проверяет, что `build/bench_mpmc` существует и исполняем.
6. Читает параметры запуска:
   - producers;
   - consumers;
   - capacity;
   - duration.
7. Пишет `run_meta.txt`.
8. Запускает сам benchmark и сохраняет `bench.csv`.
9. Запускает `perf stat` и сохраняет `perf_stat.txt`.
10. Запускает `perf record` и сохраняет `perf.data`.
11. Опционально строит cross-run графики.

### Почему нужен `run_meta.txt`

Он сохраняет метаданные сценария рядом с сырыми артефактами. Потом `plot_profiling_runs.py` может восстановить параметры запуска даже без анализа имени директории.

### Блок benchmark

Команда:

- запускает `bench_mpmc`;
- вывод одновременно сохраняется через `tee` в `bench.csv`.

Нюанс:

- затем результат копируется еще и в `${OUT_DIR}/bench_4p4c.csv`.

Это имя историческое и не зависит от реальных `P/C`, поэтому семантически немного вводит в заблуждение: файл может содержать не только `4p4c`.

### Блок `perf stat`

Собирает счетчики:

- `cycles`
- `instructions`
- `branches`
- `branch-misses`
- `cache-references`
- `cache-misses`

То есть скрипт меряет и производительность, и микроархитектурные симптомы.

### Блок `perf record`

Собирает call stacks:

- частота `999 Hz`;
- `--call-graph dwarf`.

Это уже данные не про aggregate counters, а про горячие функции и распределение CPU samples.

### Переменная `PROFILE_SKIP_PLOTS`

Если `PROFILE_SKIP_PLOTS=1`, итоговая cross-run агрегация пропускается.

Это важно при массовом цикле прогонов, когда строить общие графики после каждого run слишком дорого.

### Роль файла

Этот скрипт превращает "запуск benchmark" в "полноценный performance experiment с артефактами".

## `scripts/plot_profiling_runs.py`

### Назначение

Самый насыщенный аналитический скрипт репозитория. Он агрегирует множество profiling-run директорий и превращает их в:

- таблицы;
- summary plots;
- классификацию hot symbols.

### `@dataclass ProfilingRun`

Описывает один profiling-run:

- параметры сценария;
- throughput;
- hardware counters;
- elapsed time.

### Свойства `branch_miss_rate`, `cache_miss_rate`, `ipc`

Это вычисляемые метрики поверх сырых счетчиков:

- `branch_miss_rate = branch_misses / branches`
- `cache_miss_rate = cache_misses / cache_references`
- `ipc = instructions / cycles`

Именно они удобны для визуального анализа.

### `_parse_int(text)`

Убирает все нецифровые символы и превращает строку в integer.

Нужно, потому что `perf stat` может форматировать числа с пробелами, запятыми, локальными разделителями.

### `_parse_metric_counter(line, metric_name)`

Пытается вытащить из строки `perf stat` значение конкретного счетчика.

Это вспомогательная функция разбора неструктурированного текстового вывода `perf`.

### `parse_meta(path)`

Читает `run_meta.txt` и возвращает словарь ключ-значение.

### `parse_bench(path)`

Читает `bench.csv` и возвращает первую строку данных.

Скрипт ожидает, что в `bench.csv` один benchmark-case.

### `parse_perf_stat(path)`

Парсит `perf_stat.txt`.

Извлекает:

- cycles;
- instructions;
- branches;
- branch misses;
- cache references;
- cache misses;
- elapsed seconds.

Это критически важный мост между сырым выводом `perf stat` и дальнейшей аналитикой.

### `load_runs(runs_dir)`

Проходит по всем поддиректориям profiling-run'ов и собирает список `ProfilingRun`.

Пропускает run'ы, у которых нет полного набора файлов.

### `classify_symbol(symbol)`

Эвристическая классификация функции в одну из категорий:

- `dequeue_path`
- `enqueue_path`
- `backoff_spin`
- `atomics_control`
- `buffer_access`
- `other_code`

Это не идеальный static analysis, а rule-based группировка по подстрокам имени символа. Но для обзорного performance-анализа этого часто достаточно.

### `parse_top_symbols(perf_data_path, top_n=6)`

Что делает:

1. Вызывает `perf report --stdio`.
2. Ищет строки, относящиеся к `bench_mpmc`.
3. Парсит процент samples.
4. Классифицирует символы через `classify_symbol`.
5. Суммирует проценты по категориям.

Именно эта функция делает переход от "сырых стеков" к "распределению CPU времени по крупным зонам алгоритма".

### `latest_runs_by_pair(runs)`

Сохраняет для каждой пары `(producers, consumers)` последний встреченный profiling-run.

Нужно для summary-grid, где на клетку `P/C` нужен один актуальный run.

### `collect_symbol_maps_by_pair(runs_dir, top_n=6)`

Для каждой пары `(P, C)`:

- находит `perf.data`;
- достает классифицированное распределение символов;
- добавляет остаток в категорию `Other`;
- собирает полный список категорий.

Результат потом используется и для CSV, и для графика.

### `export_table(runs, out_csv)`

Пишет агрегированную табличку по всем run'ам.

Колонки включают:

- параметры сценария;
- throughput;
- сырые counters;
- вычисленные ratios.

Именно такой файл уже лежит в репозитории как пример:

- `docs/images/profiling_runs_summary.csv`

### `export_symbol_distribution_table(runs_dir, out_csv, top_n=6)`

Пишет таблицу вида:

- `producer, consumer, category1, category2, ...`

Это уже "сводка горячих зон кода", а не базовых counters.

Пример такого файла:

- `docs/images/profiling_symbol_distribution_table.csv`

### `plot_summary_pc_grid(runs, out_path)`

Строит большую сетку subplot'ов:

- строки - producers;
- столбцы - consumers;
- в каждой клетке маленькая гистограмма из 4 метрик:
  - throughput;
  - IPC;
  - cache miss rate;
  - branch miss rate.

Это очень компактная визуализация performance surface.

### `plot_symbol_distribution_pc_grid_subplots(runs_dir, out_path, top_n=6)`

Строит сетку, где в каждой клетке показано распределение CPU samples по категориям кода.

Это уже не "насколько быстро", а "куда уходит CPU".

### `main()`

Полный пайплайн:

1. Читает CLI-аргументы.
2. Проверяет наличие `runs/`.
3. Загружает profiling-run'ы.
4. Если данных нет, завершает работу.
5. Генерирует:
   - `profiling_runs_summary.csv`
   - `profiling_runs_summary.png`
   - `profiling_symbol_distribution_pc_grid_subplots.png`
   - `profiling_symbol_distribution_table.csv`

### Почему этот файл особенно важен

Он делает проект не просто "очередью с тестом", а именно небольшим performance-lab для исследования поведения алгоритма.

## `docs/images/profiling_symbol_distribution_table.csv`

### Что это за файл

Это агрегированная таблица распределения CPU samples по смысловым категориям кода.

### Что означают колонки

- `producer`, `consumer` - координаты сценария;
- `dequeue_path` - сколько samples ушло в логику dequeue;
- `enqueue_path` - сколько samples ушло в логику enqueue;
- `atomics_control` - вклад атомарных/контрольных операций;
- `other_code` - код, который не попал в более узкие категории;
- `backoff_spin` - время в backoff / pause;
- `buffer_access` - доступ к буферу;
- `Other` - остаточный процент.

### Практический смысл

Файл отвечает на вопрос:

- какой участок алгоритма доминирует в каждом режиме `P/C`.

Например:

- если `enqueue_path` высок при больших producer и малом числе consumer, узким местом становится вставка;
- если доминирует `dequeue_path`, значит нагрузка потребления тяжелее;
- рост `backoff_spin` и `atomics_control` обычно сигнализирует об усилении contention.

### Почему этот файл интересен

Он переводит низкоуровневый вывод `perf report` в более понятную инженерную модель.
